> **Windows/AD Focus:** An event ID is a lookup-table entry until you understand the protocol that produced it. This post builds the protocol first, then the event IDs fall into place as a story instead of a list.

**TL;DR:** <Term concept="kerberos">Kerberos</Term> replaces repeated password checks with time-limited tickets: prove who you are once to a <Term concept="kdc">KDC</Term> (event 4768), get a ticket for each service you touch afterward (event 4769), and access is granted or denied at the target without the domain controller being asked again. <Term concept="kerberoasting">Kerberoasting</Term> abuses that design directly: any domain user can request a ticket for any service account and crack it offline, no interaction with the target required. <Term concept="ntlm">NTLM</Term> is older, weaker, and still very much alive as a fallback protocol, which is exactly what makes <Term concept="ntlm-relay">NTLM relay</Term> chains like coercion-to-AD-CS still work against most enterprise networks in 2026 despite Microsoft's ongoing deprecation push. None of the event IDs below mean anything without the right audit policy turned on and a way to get them off the endpoint, which is the part most teams skip.

## Introduction

Active Directory authentication produces more useful signal, per byte logged, than almost any other data source in a Windows environment, and it's also one of the most commonly misread. An analyst who's memorized "4769 means Kerberoasting" without understanding what a service ticket actually is will chase every RC4 ticket request as an incident and still miss the pattern that actually matters. This post builds the two protocols from the ground up, Kerberos first because it's the default, then NTLM as the fallback that refuses to fully go away, and maps each concept to the specific event ID a SOC analyst will see it as. It closes with where these events physically live and what has to be configured before any of them get written at all.

## How Kerberos Actually Works

Kerberos replaces "prove your password to every server you talk to" with a three-step ticket exchange, and each step has a name and a corresponding event ID on the domain controller.

**Step one: Authentication Service Exchange (AS-REQ / AS-REP).** The client proves its identity once to the <Term concept="kdc">Key Distribution Center</Term>, the service running on every domain controller that issues and validates tickets. If the credentials check out, the KDC returns a Ticket Granting Ticket (TGT), a sealed proof of identity the client will reuse for the rest of its session instead of sending a password again. This exchange generates **event 4768**, logged on the domain controller every time a TGT is issued.

**Step two: Ticket Granting Service Exchange (TGS-REQ / TGS-REP).** When the client wants to access a specific service, say a file server or a SQL instance, it presents its TGT back to the KDC and asks for a ticket scoped to that one service, identified by its Service Principal Name (SPN). The KDC issues a service ticket encrypted with a key derived from the target service account's password, not the requesting user's. This generates **event 4769** on the domain controller, and it's the single most important Kerberos event for a SOC to understand, because it's both the normal, constant background noise of everyday AD usage and the event Kerberoasting rides on.

**Step three: Application Server Exchange (AP-REQ / AP-REP).** The client presents the service ticket directly to the target server, which decrypts it with its own password-derived key and grants or denies access locally, without contacting the domain controller again. This is why a successful logon at the target machine shows up there as **event 4624** (logon type will typically be 3, network logon, for service access) while the domain controller only ever saw the ticket request.

Tickets don't last forever. A **renewal** of an existing TGT rather than a full re-authentication generates **event 4770**.

## Kerberos vs. NTLM: What Actually Changes

<Term concept="ntlm">NTLM</Term> predates Kerberos and is still Windows' fallback protocol, used whenever Kerberos can't be: no line of sight to a domain controller, access by raw IP address instead of hostname, authentication into a workgroup machine outside any domain, or an application that simply never learned to speak Kerberos. NTLM is challenge-response: the server sends a challenge, the client responds with a value computed from the challenge and the account's password hash, and (in a domain) a domain controller verifies that response over Netlogon. Critically, the NTLM hash itself is what gets used to compute that response, which means possessing the hash is functionally equivalent to possessing the password for authentication purposes, the basis of pass-the-hash attacks.

| | Kerberos | NTLM |
|---|---|---|
| Model | Time-limited tickets, issued once and reused | Challenge-response, repeated per connection |
| Mutual authentication | Yes, client can verify the server too | No, only the server verifies the client |
| DC involved per connection | Only for the initial ticket requests | Every authentication typically round-trips to a DC |
| Credential material transmitted | Never the password or its hash directly | Response is computed from the hash; the hash itself remains password-equivalent |
| Delegation | Built-in and constrained/protected variants available | Effectively unusable for delegation |
| Offline crackability | Depends on encryption type; RC4-encrypted tickets are fast to crack, AES is far more resistant | Captured hashes and relayed sessions are directly usable, no cracking needed for relay |
| 2026 status | Default and actively hardened (RC4 deprecation underway) | Actively being deprecated, but NTLMv2 remains permitted indefinitely |

The security gap isn't really "Kerberos good, NTLM bad" as abstract properties, it's that NTLM's design makes stolen or relayed authentication material immediately and directly usable, while Kerberos at least forces an attacker through ticket theft, ticket forgery, or offline cracking, each of which is a narrower and more detectable path.

## Kerberoasting: Abusing the Protocol's Own Design

<Term concept="kerberoasting">Kerberoasting</Term> isn't a Kerberos vulnerability in the traditional sense, it's a direct consequence of how the protocol is supposed to work. Any authenticated domain user, with zero elevated privilege, is allowed to request a service ticket (a TGS, event 4769) for any account with a registered Service Principal Name, because that's precisely what the ticket exchange is for: any client should be able to ask for a ticket to a service it wants to use. The catch is what that ticket is encrypted with: a key derived from the target service account's own password, not the requester's. An attacker can request tickets for every SPN-registered account in the domain, walk away with a pile of encrypted blobs, and crack them completely offline, with no further interaction with the target accounts and no risk of triggering a lockout.

The encryption type is what makes this fast or slow. Service tickets encrypted with RC4-HMAC show up in event 4769 with a Ticket Encryption Type field value of **0x17**, and RC4 keys are derived directly from the account's NTLM password hash with no salting, which makes offline cracking dramatically faster than AES (encryption types 0x11 and 0x12). A weak or old service account password behind RC4 can fall in hours; the same password behind AES with modern hardware is a much harder proposition.

**Detection** comes down to two patterns in event 4769: RC4-HMAC (0x17) ticket requests in an environment that's supposed to be running AES, and bursts of ticket requests spanning many different SPNs from a single requesting account in a short window, which is what an automated Kerberoasting scan looks like and normal usage never does. Not every RC4 ticket is an attack; legacy applications and old service account configurations still generate them legitimately, so the signal is strongest when it's paired with account sensitivity and burst behavior rather than treated as a standalone alert. Microsoft's own 2024 guidance on mitigating Kerberoasting centers on the same two levers: move service accounts to AES-only encryption (and rotate the password afterward, since the AES key itself still derives from whatever password was set under RC4), and prefer Group Managed Service Accounts, whose passwords are long, random, and rotated automatically, making offline cracking impractical regardless of encryption type.

## NTLM Vulnerabilities to Watch For

NTLM's persistence as a fallback protocol is exactly what keeps a specific attack chain alive: **coercion, relay, and certificate abuse.**

**Coercion** techniques force a Windows machine, often a domain controller, to authenticate to a server the attacker controls. PetitPotam (CVE-2021-36942), disclosed in 2021 and still functionally relevant, abuses the Encrypting File System Remote Protocol (MS-EFSRPC) to trigger exactly this: an unauthenticated request over the LSARPC or EFSRPC named pipe causes the target to initiate NTLM authentication toward an address the attacker supplies.

**Relay** takes that captured NTLM authentication attempt and forwards it live to a second target instead of cracking anything, authenticating as the coerced identity against whatever the relay destination accepts NTLM for.

**AD CS web enrollment** is frequently that destination. Relaying a coerced domain controller's NTLM authentication to Active Directory Certificate Services' web enrollment endpoint (the attack path known as ESC8) can mint a valid certificate for that DC's identity. A certificate for a domain controller is equivalent to compromising the domain: it can be used for Kerberos PKINIT authentication, ticket forgery, and DCSync, all without ever needing a password.

This chain isn't historical. Independent 2026 research tracking exploitation in the wild found the coercion-to-relay-to-AD-CS chain still functional against the majority of enterprise environments still running Server 2019 and 2022 with AD CS deployed, which is most of them. Separately, Check Point Research documented active exploitation of **CVE-2025-24054** starting within eight days of patch release: a maliciously crafted `.library-ms` file that leaks NTLM hashes merely by a user right-clicking or navigating to the folder containing it, no execution required, with observed campaigns targeting government institutions in Poland and Romania.

Microsoft's response is a three-phase deprecation plan. Phase one, already shipped in Windows 11 24H2 and Windows Server 2025, adds enhanced NTLM auditing. Phase two, targeted for the second half of 2026, introduces IAKerb and a local KDC specifically to remove the most common reasons applications fall back to NTLM in the first place. Phase three, targeted for October 2026, blocks NTLMv1 SSO by default. Read that last point carefully: **NTLMv2, the version actually used in relay attacks, remains permitted indefinitely.** The roadmap reduces NTLM's footprint; it does not remove the relay attack surface this section describes.

## Where These Events Actually Live: EVTX, Channels, and Audit Policy

Every event ID in this post is worthless if the event never gets written or never leaves the box it was written on.

Windows stores event logs in the <Term concept="evtx">EVTX</Term> binary XML format, one file per channel, typically under `C:\Windows\System32\winevt\Logs`. The four classic channels are Application, Security, Setup, and System, but most of the events this post covers actually live in one of two places: the Security channel, or a component-specific channel under the much larger "Applications and Services Logs" tree.

| 8003 | Microsoft-Windows-NTLM/Operational | Network security: Restrict NTLM > Audit Incoming NTLM Traffic / Outgoing NTLM traffic to remote servers |

The Object Access category is the one that trips teams up most often, because 4657 and 4663 will simply never fire, silently, if a SACL was never placed on the file or registry key in question, even with the audit subcategory enabled domain-wide. Auditing a sensitive credential file or a registry run key requires both pieces: the policy that says "log object access" and the SACL on that specific object that says "log access to *this* object." Checking the policy alone and assuming coverage is one of the most common gaps a log-source audit finds.

None of this matters at SOC scale sitting on the endpoint. Windows Event Forwarding (WEF/WEC) or a log shipping agent has to move these events off the box and into centralized storage before an analyst can search across more than one machine at a time, which is the same data-centralization argument this blog has made about NSM and CSM sources generally: visibility that only exists locally, one endpoint at a time, isn't visibility a SOC can actually use.

## Sysmon: Filling the Gaps Native Logging Leaves

Even with every Advanced Audit Policy subcategory enabled, native Windows logging has real blind spots. Process creation (event 4688) doesn't include the full command line unless a separate policy is turned on, carries no file hash, and Windows logs nothing at all about outbound network connections or DNS queries. <Term concept="sysmon">Sysmon</Term>, a free Sysinternals driver and service, is how most SOCs close that gap, and it's worth understanding on its own terms because it doesn't live in the Security channel at all: it writes to Microsoft-Windows-Sysmon/Operational, and what it logs is controlled by an XML configuration file rather than Group Policy audit settings.

The Sysmon event IDs most relevant to everything covered in this post: **event 1** (process creation) captures the full command line and file hashes, catching Kerberoasting tooling like Rubeus even when it's a compiled binary and never touches PowerShell (so 4104 never fires). **Event 3** (network connection) is the only native source that sees a raw TCP connection to port 88 (Kerberos) or 445 (SMB, the transport for the NTLM relay chain described earlier), since the Security log has no equivalent. **Event 10** (ProcessAccess) is the standard detection for credential dumping via LSASS access, frequently a companion technique to NTLM hash theft. **Events 17 and 18** (named pipe created / connected) are exactly what would surface a PetitPotam-style coercion attempt riding over the `\pipe\lsarpc` or `\pipe\efsrpc` named pipes described earlier in this post, activity the Security log has no visibility into at all. Most teams don't write a Sysmon configuration from scratch; SwiftOnSecurity's community-maintained baseline config is the common starting point, tuned down from there to cut noise specific to the environment.

## The Linux Side: auditd and Syslog Facilities

Most SOCs aren't Windows-only, and the Linux equivalent of everything this post has covered exists, it just uses a completely different vocabulary. <Term concept="auditd">auditd</Term>, the Linux Audit Framework, is conceptually the closest match to Windows' Security log plus SACL model: it hooks the kernel directly, intercepting syscalls and file access rather than parsing anything after the fact. Rules are defined with `auditctl`, either at the command line or persisted under `/etc/audit/rules.d/`, and come in two flavors that map cleanly onto concepts already covered here. A **file watch** (`-w /etc/shadow -p wa -k identity`) is the rough equivalent of placing a SACL on a specific object to make event 4663 fire. A **syscall rule** (`-a always,exit -F arch=b64 -S execve -k exec`) is the rough equivalent of Sysmon's event 1, logging every process execution matching the rule. Everything lands in `/var/log/audit/audit.log` as a set of correlated record types (SYSCALL, EXECVE, PATH, CWD, tied together by a shared event ID and timestamp), queried with `ausearch` for individual events and summarized with `aureport`.

One gap worth naming explicitly: a stock Linux install ships with auditd doing very little until an administrator writes rules for it, there's no equivalent of Windows' baseline logon auditing being on by default in most hardening guides. The closest thing Linux has to "who logged on and how, by default" isn't auditd at all, it's standard syslog, and specifically the **auth**/**authpriv** <Term concept="syslog-facility">facility</Term>, the channel SSH, `sudo`, and PAM write to whenever someone authenticates or elevates privilege. Facility (what subsystem generated the message) plus severity is what a log router like rsyslog or syslog-ng uses to decide where a message ends up, and `auth,authpriv.* /var/log/secure` (or `/var/log/auth.log` on Debian-family systems) is the classic rule capturing Linux's rough answer to what 4624 and 4768 answer on a domain controller.

| Concept | Windows | Linux |
|---|---|---|
| Kernel/native security log | Security.evtx, gated by Advanced Audit Policy | auditd, gated by explicit auditctl rules |
| Rich endpoint telemetry (process, network, pipes) | Sysmon (Microsoft-Windows-Sysmon/Operational) | auditd syscall rules, often paired with eBPF-based tooling |
| Default logon/auth visibility | Event 4624, on in most baselines | auth/authpriv syslog facility, SSH and PAM |
| Query tooling | Event Viewer, `wevtutil`, PowerShell `Get-WinEvent` | `ausearch`, `aureport` |
| Object-level watch mechanism | SACL on the specific object | `auditctl -w` file watch rule |

The mechanics differ enough that a SOC covering both platforms genuinely needs to learn two vocabularies, but the underlying question stays identical on either side: who did what, to what, and when, with enough default coverage turned on that the answer exists before it's needed.

## The Full Event ID Reference

Grouped by what they actually tell you, rather than numeric order.

**Kerberos and logon (the authentication story itself)**
- **4768**: TGT requested (initial logon to the domain)
- **4769**: Service ticket requested (access to a specific SPN; watch encryption type for Kerberoasting)
- **4770**: Service ticket renewed
- **4624**: Successful logon (check the Logon Type: 3 is network, 10 is RDP, 2 is interactive)

**Account and group changes (privilege and persistence via identity)**
- **4720**: User account created
- **4732**: Member added to a security-enabled local group (watch local Administrators)
- **4728**: Member added to a security-enabled global group

**Object and registry access (data and configuration touched)**
- **4657**: Registry value created, modified, or deleted (requires a SACL on the key)
- **4663**: An object handle was used to exercise specific access rights (requires a SACL on the object)

**Persistence and execution (how an attacker stays and runs code)**
- **7045**: New service installed (classic PsExec-style lateral movement and persistence signal)
- **4698**: Scheduled task created
- **4104**: PowerShell script block logged (the single highest-value source for catching obfuscated PowerShell)

**Device and endpoint signals**
- **6416**: New external device recognized (USB media; alert when the Subject isn't SYSTEM)
- **1006 / 1116**: Defender detected malware or unwanted software (1006 on the legacy engine, 1116 on the modern Windows Defender/Operational channel)

**NTLM auditing (finding out what still depends on the protocol before Microsoft turns it off)**
- **8003**: Incoming NTLM authentication logged on a member server, generated once NTLM auditing is enabled; the practical first step of any NTLM reduction project is collecting two weeks of these and building an inventory of what's still asking for NTLM before restricting it

## Conclusion

Kerberos gives a SOC a clean story, in principle: 4768 says who authenticated, 4769 says what they asked to access next, and 4624 confirms whether the target actually let them in, all tied together by the same client and the same window of time. NTLM doesn't tell that story nearly as cleanly, which is precisely why an attacker forcing a target back onto NTLM, through coercion, through an application that never learned Kerberos, through a stale GPO nobody re-audited, is choosing the protocol that's harder to reason about, not just the one that's easier to relay. None of the event IDs in this post substitute for that reasoning. They're the vocabulary; the protocol is the grammar that makes the vocabulary mean something.
